## 背景      
### 11.3.4 数据保留与压缩  
  
在时序数据库应用中，数据量增长速度极快，例如物联网设备每天产生数亿条记录、金融市场每秒产生数百万笔交易。随着数据累积，如果不进行合理管理，存储成本和查询性能都会急剧下降。  
  
TimescaleDB 提供了 数据保留策略（Data Retention Policies） 和 压缩机制（Compression），使数据库能够在保持可查询性的同时，降低存储开销并优化查询性能。  
  
   
  
#### 一、数据保留策略（Retention Policies）  
  
数据保留策略的核心目标是：  
  
* 控制旧数据占用的存储空间  
* 满足业务对历史数据的保留要求  
* 与连续聚合结合实现冷热数据分离  
  
TimescaleDB 提供了 后台任务（Background Job） 支持自动删除过期数据。  
  
   
  
##### 1. 基本保留策略  
  
假设业务需求：保留最近 90 天的原始数据，删除过期数据。  
  
创建 Hypertable：  
  
```sql id="t1f7ds"  
CREATE TABLE cpu_metrics (  
    time TIMESTAMPTZ NOT NULL,  
    host TEXT,  
    cpu_usage DOUBLE PRECISION  
);  
  
SELECT create_hypertable('cpu_metrics','time');  
```  
  
添加自动删除策略：  
  
```sql id="k8l2qz"  
SELECT add_retention_policy(  
    'cpu_metrics',  
    older_than => INTERVAL '90 days'  
);  
```  
  
该策略会在后台定期运行：  
  
* 删除超过 90 天的数据 chunk  
* 自动释放磁盘空间  
* 避免人工管理  
  
> 注意：删除 chunk 的粒度是时间分片（Chunk），因此选择合适的 chunk interval 至关重要。  
  
   
  
##### 2. 分片粒度与删除策略  
  
Chunk interval 决定每个时间段存储多少数据：  
  
| chunk_interval | 优势        | 劣势               |  
| -------------- | --------- | ---------------- |  
| 1 天            | 删除/压缩粒度精细 | Chunk数量多，管理开销大   |  
| 7 天            | 管理简单      | 删除精度较低，可能删除过多或不足 |  
  
例如：  
  
```sql id="p8gzqf"  
SELECT create_hypertable(  
    'cpu_metrics',  
    'time',  
    chunk_time_interval => INTERVAL '1 day'  
);  
```  
  
这种设计可保证每天产生的数据存储在单独 chunk 中，便于按日管理。  
  
   
  
#### 二、数据压缩（Compression）  
  
时序数据通常具有高度重复性和低基数字段，非常适合压缩。  
  
TimescaleDB 压缩机制特点：  
  
* 列式压缩：将列拆分存储，重复值仅存一次  
* 块级压缩：按 chunk 压缩  
* 支持解压查询：查询时自动解压数据  
  
官方测试显示：  
  
* 温度传感器数据压缩比可达到 10:1 ~ 20:1  
* 金融行情数据压缩比可达到 5:1 ~ 10:1    
  
   
  
##### 1. 启用压缩  
  
启用压缩前，需要先设置 hypertable 支持压缩：  
  
```sql id="r0s7ak"  
ALTER TABLE cpu_metrics SET (  
    timescaledb.compress,  
    timescaledb.compress_segmentby = 'host'  
);  
```  
  
参数说明：  
  
* `timescaledb.compress`：启用压缩  
* `timescaledb.compress_segmentby`：分组字段，通常选低基数字段，例如 host 或 device_id  
  
   
  
##### 2. 执行压缩  
  
压缩数据通常基于 chunk 进行：  
  
```sql id="m4t8nl"  
SELECT compress_chunk(i)   
FROM show_chunks('cpu_metrics') i  
WHERE i < now() - INTERVAL '30 days';  
```  
  
逻辑：  
  
* 压缩 30 天前的数据  
* 保留最近 30 天数据未压缩以支持高速写入  
* 压缩后查询仍然透明，SQL 无需修改  
  
> 注意：压缩主要面向 历史数据，保证实时写入性能不受影响。  
  
   
  
##### 3. 压缩查询示例  
  
压缩后查询无需关心底层存储：  
  
```sql id="c2y7x0"  
SELECT time_bucket('1 hour', time) AS hour,  
       avg(cpu_usage) AS avg_cpu  
FROM cpu_metrics  
WHERE time > now() - INTERVAL '60 days'  
GROUP BY hour;  
```  
  
数据库会自动解压所需数据进行聚合，保证 透明性和兼容性。  
  
   
  
#### 三、冷热数据管理  
  
通过数据保留和压缩策略，TimescaleDB 可以实现 冷热数据分离：  
  
| 数据类型 | 存储策略       | 查询特点        |  
| ---- | ---------- | ----------- |  
| 热数据  | 未压缩、最近 7 天 | 高吞吐写入、低延迟查询 |  
| 冷数据  | 压缩、历史数据    | 低成本存储、分析查询  |  
  
这种分层设计保证：  
  
* 实时分析高效  
* 历史数据存储经济  
  
   
  
#### 四、企业应用案例  
  
##### 1. IoT 设备监控  
  
* 数据量：每台设备每秒 1 条  
* 数据量大：百万设备 → 数十亿条/天  
* 策略：最近 7 天数据热存储，其余数据压缩存储，保留 1 年历史数据  
  
   
  
##### 2. IT 运维监控  
  
* 服务器 CPU、内存、网络指标  
* 保留策略：最近 90 天详细数据  
* 历史数据压缩存储以支持长期趋势分析  
  
   
  
##### 3. 金融交易系统  
  
* 高频交易数据量极大  
* 数据保留策略：按日存储  
* 超过 1 年的数据压缩存储，用于回测和趋势分析  
  
   
  
#### 小结  
  
数据保留与压缩是 大规模时序数据库的核心能力，直接影响存储成本和查询性能。  
  
| 功能      | TimescaleDB实现方式                     |  
| ------- | ----------------------------------- |  
| 数据保留    | `add_retention_policy` 自动删除过期 chunk |  
| 压缩      | Chunk级列式压缩，支持 segment-by 分组         |  
| 热/冷数据分层 | 最近数据未压缩，历史数据压缩存储                    |  
| 查询透明性   | SQL 查询无需修改，自动解压历史数据                 |  
  
通过合理的数据保留和压缩策略，TimescaleDB 可以在 PB级时序数据场景下保持高效存储和实时分析能力，同时延长硬件寿命、降低存储成本。
