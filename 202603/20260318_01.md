## PostgreSQL 19 preview - PG 狂抠 CPU 指令级性能, 今天开始 deformation 少读 8 字节 
              
### 作者              
digoal              
              
### 日期              
2026-03-18              
              
### 标签              
PostgreSQL , CompactAttribute , tuple deformation , CPU 指令级      
              
----              
              
## 背景   
2026 年 3 月 17 日，PostgreSQL 提交了一条很多 DBA 第一眼会直接划走的 commit：[`d8a859d22b1accd8ea749394a488e4de014b2396`](https://git.postgresql.org/gitweb/?p=postgresql.git;a=commit;h=d8a859d22b1accd8ea749394a488e4de014b2396)，标题很朴素：**把 `CompactAttribute` 从 16 字节压到 8 字节**。  
  
如果你觉得这只是“内核开发者闲得精细化抠字节”，那我先把观点摆明白：  
  
**这类优化不是小修小补，而是 PostgreSQL 在 CPU 时代对执行器热路径的一次正面进攻。**  
真正危险的，不是你没看懂这个 commit；而是你还在用“磁盘快不快”解释一切数据库性能问题。  
  
    
  
## **一、为什么 DBA 必须重视这 8 字节？**  
  
因为它打的不是冷门代码，而是 `tuple deformation`。  
  
说人话：PostgreSQL 从 heap tuple 里把列值“拆出来”供执行器使用，这个过程几乎存在于所有常规查询执行路径里。只要查询要取列、算表达式、做过滤、输出结果，就绕不开这一层。DeepWiki 对 PostgreSQL 源码路径的总结也明确指出，`CompactAttribute` 被放在 `TupleDesc` 里，就是为了让 `heap_deform_tuple`、`index_deform_tuple`、`fastgetattr` 这类高频路径少碰 `FormData_pg_attribute` 这种更重的元数据结构。  
  
这意味着什么？  
  
**意味着它不是优化某个 SQL 语法角落，而是在优化“每行、每列、每次取值”的基础动作。**  
  
数据库性能里，凡是进入“按元组、按属性、按循环次数”计费的代码，都不能用“只差几个字节”来轻视。因为一旦进入热路径，单位成本乘以调用次数，最后就是整机 CPU 账单。  
  
   
  
## **二、第一性原理：这条 commit 到底在优化什么？**  
  
核心不是“结构体变小”本身，而是下面三件事：  
  
1. **更少的缓存行访问**  
2. **更高的元数据密度**  
3. **更便宜的地址计算**  
  
提交说明写得很直白：`CompactAttribute` 以前是 16 字节，现在通过 bit flags 和把 `attcacheoff` 缩成 16 位，减半到 8 字节。这样做的收益主要有两个：  
  
- tuple deformation 过程中，要访问的属性元数据能落进更少的 cachelines  
- 在 x86 上，8 字节步长更利于用一条 `LEA` 指令完成地址计算  
  
这不是玄学，这是典型的微架构收益。  
  
**推导一下就更直观了。**  
在常见 64B cache line 上，16B 结构体一行只能装 4 个属性；8B 结构体一行能装 8 个属性。    
这意味着在“宽表 + 多列解包”的情况下，**同样一次 cache miss，能带回双倍属性元数据**。这部分虽然是我基于缓存行常识做的推导，但方向和 commit 本身强调的“fewer cachelines”完全一致。  
  
   
  
## **三、别忘了背景：这不是孤立优化，而是一条持续路线**  
  
如果你只看这一个 commit，很容易误判成“过度优化”。  
  
但实际上，PostgreSQL 社区在这条线上已经持续推进很久了。  
  
2024 年 12 月，PostgreSQL 先引入了 `CompactAttribute`，把 `TupleDesc` 里每列常用元数据从原本 `FormData_pg_attribute` 的 **104 字节**，压缩到 **16 字节**，目的是让 tuple deformation 等关键路径不必反复碰大对象元数据。[提交记录在这里](https://www.postgresql.org/message-id/E1tIJx5-000qa3-44%40gemulon.postgresql.org)  
  
现在这条 2026 年 3 月 17 日的 commit，本质上是**第二刀**：    
**16 字节还不够小，继续砍到 8 字节。**  
  
这说明什么？  
  
说明 PostgreSQL 核心开发者的判断非常明确：  
  
**执行器元数据访问，已经是值得用“结构体布局”和“CPU 指令级别”去抠的瓶颈。**  
  
这不是拍脑袋。邮件讨论里，Andres Freund、Álvaro Herrera、Chao Li 等核心开发者都围绕 tuple deformation 的代码生成、缓存行为、benchmark 稳定性反复讨论。甚至 Andrés 还给出过一个很扎眼的数据：某个循环在编译器错误向量化/展开后，指令数从大约 26 条膨胀到 135 甚至 532 条，说明**代码形态本身就会直接改变 CPU 执行成本**。[讨论线索](https://www.postgresql.org/message-id/v6z545yozjtywghn5glujemu72z4i4ynadsc2xks4ejotdg7yl%404rry7ixwr4us)  
  
更重要的是，Chao Li 在 2026 年 1 月 23 日的测试中提到，v4 patch 在 MacBook M4 上，**最佳场景下比 master 快约 43%**；额外 tweak 还能再带来约 **3.5%** 的提升。[来源](https://www.postgresql.org/message-id/82AD055C-3280-4DFB-ADA8-A7A4DE3844A5%40gmail.com)  
  
请注意，我没有说“这一个 8 字节 commit 就能让你的业务快 43%”。    
严谨地说，**43% 来自同一优化方向的一组补丁测试，而不是单个 commit 的公开官方结论**。但它足够说明一件事：  
  
**tuple deformation 这条路径，确实存在可观的 CPU 优化空间。**  
  
   
  
## **四、对 DBA 最重要的判断：哪些场景会真正受益？**  
  
这里必须讲前提，不讲前提就是标题党。  
  
**这条 commit 更可能有明显价值的前提条件：**  
  
- 你的负载已经不是 I/O 主导，而是 **CPU 主导**  
- 数据大部分在共享缓冲区和 OS page cache 里，**不是磁盘 miss 主导**  
- 查询会频繁读取 heap tuple，并做列提取、表达式计算、投影输出  
- 表不算特别窄，或者查询涉及较多列访问  
- 执行器在高 QPS 下反复走同一类解包逻辑  
  
在这些前提下，这类元数据缩容优化是有现实意义的。因为你的瓶颈已经从“能不能把数据读出来”，变成了“CPU 用多高的单位成本处理每一行”。  
  
**反过来，如果这些条件崩塌，观点也必须调整：**  
  
- 如果系统瓶颈是磁盘 I/O、网络、锁等待、WAL 刷盘，这个 commit 的体感大概率接近于零  
- 如果业务是重度 varlena、JSONB、TOAST、函数计算，列偏移本身就不容易缓存，收益会被稀释  
- 如果查询本来就只取极少数列，且 tuple deformation 不是热点，这类优化不是主战场  
  
所以，真正专业的 DBA 不会说“这个优化没用”，也不会说“升级就起飞”。    
更准确的说法应该是：  
  
**它对“内存命中、CPU 吃紧、行处理密集”的系统更重要；对“慢在别处”的系统则只是锦上添花。**  
  
   
## **五、这条 commit 给 DBA 的真正启示，不在 8 字节，而在架构判断**  
  
很多团队今天还在用一个过时认知管理 PostgreSQL：  
  
**只要存储够快，数据库就会快。**  
  
这已经越来越不成立了。  
  
现代 PostgreSQL 的真实战场，越来越像这样：  
  
- 磁盘比过去快得多  
- 内存容量更大，热点数据更容易常驻  
- NVMe 把一部分 I/O 问题压下去后，CPU、缓存层次、分支预测、指令数开始抬头  
- OLTP 和 HTAP 混合场景下，“每行多花几个周期”会被放大成可观的整体损耗  
  
因此，这条 commit 的战略意义其实很清楚：  
  
**PostgreSQL 正在从“把功能做出来”走向“把每个 CPU 周期榨干”。**  
  
对 DBA 来说，这会带来两个现实结论：  
  
1. **升级版本的收益，不再只是新特性。**    
很多收益会来自你平时根本看不到的执行器热路径优化。  
  
2. **宽表滥用、无节制列膨胀、把数据库当 JSON 文件柜，代价会越来越清晰。**    
因为内核已经在尽力降低元数据访问成本了，如果你的模型设计仍然让每行处理成本过高，那你是在拿架构缺陷对冲内核优化。  
  
   
  
## **六、我的结论很尖锐，但我认为站得住**  
  
**PostgreSQL 的下一阶段竞争力，不只是 SQL 能力，不只是插件生态，而是 CPU 效率。**  
  
`CompactAttribute` 从 16B 砍到 8B，这种 commit 看起来不像发布会 headline，但它恰恰代表了数据库内核最有含金量的一类进步：    
**把“每行都会发生的事”做得更便宜。**  
  
对 DBA 来说，真正该记住的不是结构体名字，而是这个判断框架：  
  
- 如果你的系统还慢在 I/O，这类优化不是主角  
- 如果你的系统已经进入 CPU 热点竞争，这类优化就是实打实的生产力  
- 如果你长期运行高并发、热点常驻、列访问密集的业务，升级 PostgreSQL 版本时，千万别只盯 SQL 特性列表，**执行器微优化同样是利润表**  
  
说得再直白一点：  
  
**未来 PostgreSQL 的性能红利，越来越不是“多一条参数建议”，而是“少一次 cache miss”。**  
  
这才是 DBA 真正该看懂的方向。

Reduce size of CompactAttribute struct to 8 bytes  
  
Previously, this was 16 bytes.  With the use of some bitflags and by  
reducing the attcacheoff field size to a 16-bit type, we can halve the  
size of the struct.  
  
It's unlikely that caching the offsets for offsets larger than what will  
fit in a 16-bit int will help much as the tuple is very likely to have  
some non-fixed-width types anyway, the offsets of which we cannot cache.  
  
Shrinking this down to 8 bytes helps by accessing fewer cachelines when  
performing tuple deformation.  The fields used there are all fully  
fledged fields, which don't require any bitmasking to extract the value  
of.  It also helps to more efficiently calculate the address of a  
compact_attrs[] element in TupleDesc as the x86 LEA instruction can work  
with 8 byte offsets, which allows the element address to be calculated  
from the TupleDesc's address in a single instruction using LEA's  
concurrent shift and add.
