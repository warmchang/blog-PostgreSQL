Vastbase G100 可能很多人不知道是啥, 简单介绍一下. 它是海量的一款企业级关系型数据库，旨在提供强竞争力的事务型、集中式高可用数据库。   
- 支持 SQL2003 标准语法。  
- 对Oracle、MySQL、Postgres、SQL Server 做了大量的兼容性支持，同时，提供高效的异构数据库迁移工具exBase，实现应用平滑迁移。  
- 多种存储模式支持复合业务场景，新引入提供原地更新存储引擎。  
- Paxos一致性日志复制协议，主备模式，CRC校验支持高可用。  
- NUMA-AWARE、日志无锁并行写入、线程池、CSN事务快照技术合力构筑高性能。  
- 支持全密态计算、账本数据库等安全特性，提供全方位端到端的数据安全保护。  
- 支持向量数据存储和检索，通过特征提取和向量化技术将文本、图像、音频等不同类型的非结构化数据转换为向量并存储在Vastbase G100中，为AI应用提供了向量运算支持。  
  
现在国产数据库的文档确实越来越好了, 我在测试过程中, 完全独立完成, 没有和厂家有任何交流.  
  
下面的测试包含了五个模块(镜像加载、性能、列存储对比行存储、in-place 原地更新存储引擎、国密功能), 我也会备注用到的手册URL. 有兴趣的小伙伴可以和我一起跑一跑看看.

踩到的第一个坑:  
  
文档有一处小错误, 也可能是 macOS 中不需要这么做? 反正一旦修改了owner, 会导致权限错误, 容器无法启动  
> 容器中运行数据库的用户为 vastbase 且 uid:gid 为 1000:1000，为保证容器中有权限访问被挂载的目录，宿主机中需要将被挂载的目录和文件的属性 uid:gid 修改为 1000:1000。  
  
```  
mkdir -p ~/vastbase/license  
mkdir -p ~/vastbase/data  
mkdir -p ~/vastbase/backup  
mkdir -p ~/vastbase/backup_log  
  
# 所以不需要执行这一步  
# sudo chown -R 1000:1000 ~/vastbase  
  
# 下面的所有变量, 请从手册了解具体含义  
LICENSE_VDIR=$(cd ~/vastbase/license; pwd)  
DATA_VDIR=$(cd ~/vastbase/data; pwd)  
BACKUP_VDIR=$(cd ~/vastbase/backup; pwd)  
BACKUP_LOG_VDIR=$(cd ~/vastbase/backup_log; pwd)  
VB_USERNAME="postgres"  
VB_PASSWORD="Vbase@123"  
VB_DBCOMPATIBILITY="PG"  
  
docker run -d --privileged=true \  
-v ${LICENSE_VDIR}:/home/vastbase/vastbase/lic \  
-v ${DATA_VDIR}:/home/vastbase/data  \  
-v ${BACKUP_VDIR}:/home/vastbase/backup \  
-v ${BACKUP_LOG_VDIR}:/home/vastbase/backup_log  \  
-e VB_USERNAME=${VB_USERNAME} \  
-e VB_PASSWORD=${VB_PASSWORD} \  
-e VB_DBCOMPATIBILITY=${VB_DBCOMPATIBILITY} \  
--name g100 -p 25432:5432 \  
vastbase-g100-single-v1.4-3.0.8.29407-linux-kunpeng920:20260119165657   
```

## 5、ustore (in-place update存储引擎)  
  
https://docs.vastdata.com.cn/zh_CN/VastbaseG100/V3.0.9/1/4776d12b29fe4b7bbd3053b3f7b2c5fc  
  
Vastbase G100提供了Ustore存储引擎，引入了原地更新（In-place Update）机制，相较于Astore行存储引擎所采用的追加更新（Append Update）模式，提供了显著的优势。追加更新对INSERT、DELETE操作以及热更新（即HOT Update，指页面内的数据更新）表现出色，但在处理涉及跨数据页面的非热更新时，其垃圾回收效率较低。而Ustore通过实现原地更新，优化了这一过程，使得在面对跨页更新场景时，能够提供更加高效的数据管理与垃圾回收能力，从而提升了整体性能和资源利用率。  
  
下面来测试一下, 主要是在频繁的更新后, 膨胀的情况  
  
建表并写入数据  
  
```  
CREATE TABLE test_ustore  
(  
  id   int primary key,  
  info text,  
  ts timestamp  
)  
WITH (storage_type=ustore);
```  
