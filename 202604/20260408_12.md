## 背景  

本文解读 PostgreSQL commit d1c01b7 "aio: Adjust I/O worker pool automatically"，该 commit 将 PostgreSQL AIO 子系统中 `io_method=worker` 的 worker 池从静态配置改为自适应动态调节机制。传统的 `io_workers` 参数（默认值 3）要求 DBA 人工估算最佳 worker 数量，配置困难且无法适应负载变化。新方案引入 `io_min_workers`、`io_max_workers`、`io_worker_idle_timeout` 和 `io_worker_launch_interval` 四个参数，worker 池根据实际 I/O 队列积压情况自动伸缩——在队列积压超过当前 worker 数时"热扩展"，在最高编号 worker 空闲超时后"冷收缩"，同时通过 wakeup:IO 比值抑制虚假唤醒。

  


---

## 背景与痛点

### AIO Worker 池的作用

PostgreSQL 的 AIO（Asynchronous I/O）子系统允许数据库在执行 I/O 操作时不阻塞调用者。其中 `io_method=worker` 是一种跨平台通用实现，通过一组独立的 worker 子进程从共享内存中的提交队列（submission queue）取走 I/O 请求，执行同步系统调用（如 `pread`/`pwrite`），完成后通过条件变量通知等待的后端。

```
┌─────────────┐    共享内存提交队列     ┌──────────────────┐
│   Backend   │ ──────────────────────►│  Worker Pool    │
│  (submitter)│    PGAioWorkerSubmission│  (2~32 workers) │
│             │◄────────────────────── │  执行同步 I/O    │
└─────────────┘      完成通知           └──────────────────┘
```

在没有 AIO 的情况下，后端进程必须自己执行 I/O 并等待其完成，这会阻塞处理查询的进程，影响并发能力。Worker 池模式将 I/O 操作卸载到独立进程，让后端可以并行处理其他工作。

### 原有方案的困境

在此 commit 之前，worker 池大小由单一 GUC 参数 `io_workers` 控制（默认 3），取值范围 1~32。这是一个**静态配置**，存在以下问题：

1. **配置玄学**：DBA 很难确定"我的负载需要多少 worker"。3 可能太少，32 可能浪费。
2. **无法自适应**：业务负载有潮汐波动——白天繁忙，夜间空闲。固定 worker 数在低负载时浪费资源，在突发 I/O 时又可能成为瓶颈。
3. **扩展与收缩依赖人工**：增大 worker 数需要 reload 配置并由 postmaster 启动新 worker；减少 worker 数需要重启或等待 worker 异常退出。
4. **唤醒效率低下**：每个 worker 可以唤醒两个同伴，但这种"扇出"机制没有反馈调节，容易产生无意义的虚假唤醒（spurious wakeup），浪费 CPU。

正是针对这些痛点，该 commit 引入了**自适应 worker 池**。

## 原理介绍

### 核心设计思想

Worker 池在 `io_min_workers` 和 `io_max_workers` 范围内自动调节：

- **扩容（Grow）**：当 worker 发现提交队列深度超过当前 worker 总数时，认为可能存在排队延迟，向 postmaster 请求启动新 worker。
- **缩容（Shrink）**：池中编号最高的 worker 在空闲超时后主动退出，确保池大小不低于 `io_min_workers`。

### 新增 GUC 参数

| 参数 | 默认值 | 含义 |
|------|--------|------|
| `io_min_workers` | 2 | 池中保持的最小 worker 数 |
| `io_max_workers` | 8（最大 32） | 池中允许的最大 worker 数 |
| `io_worker_idle_timeout` | 60s | idle worker 退出前等待的时间 |
| `io_worker_launch_interval` | 100ms | 两次 worker 启动之间的最小间隔 |

**撤销的旧参数**：`io_workers`（固定数量）被完全替换。

### 共享数据结构

每个 worker 在 `PgAioWorkerControl` 中登记，核心字段：

```c
typedef struct PgAioWorkerControl {
    bool        grow;               /* 是否请求扩容（worker→postmaster）*/
    bool        grow_signal_sent;   /* 信号是否已发送 */
    PgAioWorkerSet idle_workerset;  /* 空闲 worker 集合（按编号）*/
    PgAioWorkerSet workerset;      /* 所有在线 worker 集合 */
    int         nworkers;           /* 当前 worker 总数 */
    PgAioWorkerSlot workers[];      /* 每个 slot 含 proc_number */
} PgAioWorkerControl;
```

`PgAioWorkerSet` 是一个 64 位 bitmap，每位对应一个 worker 编号（0~31），用 `pg_leftmost_one_pos64()` 取最高位、`pg_rightmost_one_pos64()` 取最低位。这比遍历数组更高效。

### 扩容机制（Grow）

扩容的触发发生在 worker 的工作循环中（见 `method_worker.c:IoWorkerMain`）：

```c
if (hist_wakeups <= hist_ios) {
    queue_depth = pgaio_worker_submission_queue_depth();
    if (queue_depth > 0) {
        worker = pgaio_worker_choose_idle(MyIoWorkerId);
        if (worker == -1)           // 没有比当前编号更高的 idle worker
            maybe_grow = true;
    }
}
```

如果 `maybe_grow = true`，调用：

```c
static void pgaio_worker_request_grow(void)
{
    if (io_worker_control->nworkers >= io_max_workers) return; // 已达上限
    if (io_worker_control->grow) return;                       // 已请求过

    io_worker_control->grow = true;
    SendPostmasterSignal(PMSIGNAL_IO_WORKER_GROW); // 发信号给 postmaster
    io_worker_control->grow_signal_sent = true;
}
```

Postmaster 收到 `PMSIGNAL_IO_WORKER_GROW` 信号后，在 `maybe_start_io_workers()` 中根据 `io_worker_launch_interval`（默认 100ms）控制启动节奏，避免在突发 I/O 时一口气拉起大量 worker。

### 缩容机制（Shrink）

Worker 的主循环在空闲时会进入 `WaitLatch` 并带上超时：

```c
if (pgaio_worker_can_timeout()) {
    // 只有"最高编号 worker"且 pool 大于 io_min_workers 时才能超时退出
    if (GetCurrentTimestamp() >= idle_timeout_abs)
        break;  // 退出循环 → worker 进程结束
}
```

**为什么必须是最高编号？** 因为 bitmap 的特性使 worker ID 天然有序（0, 1, 2, ...），只有最高 ID 的 worker 退出才不会在 ID 序列中留下空洞，保证 pool 始终是 `0..N` 的连续编号范围，从而简化逻辑并确保不会意外跌破 `io_min_workers`。

### 虚假唤醒抑制（Spurious Wakeup Reduction）

每个 worker 维护两个计数器：

```c
int hist_ios = 0;      // 处理的 I/O 数
int hist_wakeups = 0;  // 收到的 wakeup 次数
```

当收到一次有效的 I/O 时，两者同步递增，到达饱和值（4）时都除以 2——这是一个**指数衰减**的比率估计。判断是否传播 wakeup 给同伴：

```c
if (hist_wakeups <= hist_ios) {
    // 我自己没怎么白起床，才去叫醒更高编号的同伴
    worker = pgaio_worker_choose_idle(MyIoWorkerId);
}
```

如果一个 worker 频繁被唤醒但没干多少活（`hist_wakeups > hist_ios`），说明唤醒链条已经超过了必要范围，它会停止继续唤醒更高编号的同伴，让它们继续睡觉。

### Postmaster 侧的调度

`maybe_adjust_io_workers()` 被重构为两个函数：

```c
// 返回下次应该启动 worker 的时间戳（0 表示不需要）
static TimestampTz maybe_start_io_workers_scheduled_at(void);

// 实际执行启动逻辑
static void maybe_start_io_workers(void);
```

`DetermineSleepTime()` 利用 `maybe_start_io_workers_scheduled_at()` 计算下次醒来的时机——如果 postmaster 正在等待 worker 启动，就只睡 `io_worker_launch_interval`，而不是固定的 60 秒。

```c
static TimestampTz maybe_start_io_workers_scheduled_at(void)
{
    if (io_worker_count >= io_max_workers) return 0;
    if (io_worker_count < io_min_workers) return TIMESTAMP_MINUS_INFINITY; // 立即
    if (!pgaio_worker_pm_test_grow_signal_sent()) return 0;                   // 无请求
    return io_worker_launch_next_time;                                        // 按调度
}
```

### 新增锁保护

引入了专门的 `AioWorkerControlLock`（LWLock #57），将 worker 元数据（`workerset`、`nworkers`）的保护与队列操作（`AioWorkerSubmissionQueueLock`）分离，减少锁竞争。

## 应用场景及最佳实践

### 场景一：潮汐负载——夜间批量导入

**Commit 前：**
DBA 设置 `io_workers = 8`，白天正常，但夜间批量导入时 I/O 突发导致队列积压，延迟上升。夜间空闲时 8 个 worker 仍然运行，浪费资源。

**Commit 后：**
假设配置 `io_min_workers = 2`，`io_max_workers = 8`，`io_worker_idle_timeout = 60s`：

- 白天查询高峰：队列积压触发自动扩展，worker 逐渐增加到 5~6 个
- 夜间导入突发：worker 达到 8 个上限
- 导入结束后：最高编号 worker idle 60s 后退出，pool 逐渐缩回 2 个

**效果**：无需人工干预，资源使用与负载匹配。

### 场景二： OLTP 混合负载

**Commit 前：**
`io_workers = 3` 对于 OLTP 场景够用，但当出现大量小 I/O（如多表并发扫描）时，3 个 worker 竞争同一把提交队列锁（`AioWorkerSubmissionQueueLock`），成为瓶颈。

**Commit 后：**
Worker 发现队列深度持续超过 nworkers（> 3），持续请求 grow，postmaster 在 100ms 间隔限制下逐步拉起新 worker，直到队列被消化。每次扩容都基于实测排队深度，而非经验估算。

**效果**：动态消除了因固定 worker 数导致的锁竞争。

### 场景三：虚假唤醒导致的 CPU 浪费

**Commit 前：**
即使 I/O 非常快（微秒级），worker 也可能唤醒同伴来"帮忙"，导致多个 worker 被唤醒后发现队列已空。这种虚假唤醒在大规模部署中会消耗可观的 CPU 资源。

**Commit 后：**
Worker 统计 wakeup:IO 比率，如果本身收到的 wakeup 远多于实际处理的 I/O，说明已经到了"链条末端"，停止传播。这条链会动态收敛到最优长度。

**参数调优建议**：
- `io_worker_idle_timeout = 60s`（默认）：适合稳定负载。如果业务波动剧烈，可降至 30s 更快收缩。
- `io_worker_launch_interval = 100ms`（默认）：防止突发 I/O 时过度扩展。可根据业务 I/O 突发频率调整——突发越频繁，间隔可稍大。

### 配置对比

| 场景 | io_min_workers | io_max_workers | io_worker_idle_timeout |
|------|----------------|-----------------|------------------------|
| 低负载占主导 | 1~2 | 4~6 | 30s~60s |
| 高吞吐混合负载 | 4~6 | 16~32 | 60s~120s |
| 开发测试（资源受限）| 1 | 4 | 10s |

## 小结和思考

### 核心价值

该 commit 将 PostgreSQL AIO worker 池从"人工静态配置"提升为"数据驱动的自适应系统"，解决了长期困扰 DBA 的调参难题。通过引入排队深度检测、空闲超时和唤醒比率三大反馈机制，worker 池能够：

1. **零配置开箱即用**：默认参数（2~8 workers，60s idle timeout）已经能适应大多数场景。
2. **资源按需分配**：低负载时自动收缩，高负载时自动扩张，无需人工干预。
3. **避免过度唤醒**：通过 wakeup:IO 比率限制唤醒传播链，减少无效 CPU 消耗。

### 设计亮点

- **最高编号唯一超时退出**的设计非常巧妙——用 bitmap 的天然有序性确保了 pool 编号连续不空洞，同时保证了 `io_min_workers` 硬下限。
- **`io_worker_launch_interval` 的节流设计**避免了 I/O 突发时 postmaster 被信号淹没，一步步"探测"最优 pool 大小。
- **反馈与限流结合**：grow 路径有 queue_depth > nworkers 的实测反馈；shrink 路径有 60s idle timeout 的时间窗口；两者配合防止震荡。

### 可能的演进方向

Commit 注释中提到：`XXX Perhaps ideas from queueing theory or control theory could do a better job of this`。确实，当前阈值（queue_depth > nworkers）是一个比较粗的启发式规则，未来可以考虑引入排队论模型（如 Little 定律）或自适应控制理论（如 PID 控制器）来进一步优化动态调节的响应速度和稳定性。

对于 DBA 而言，这意味着 PostgreSQL 在 I/O 资源管理上越来越"智能"，减少了人工微调的必要，同时也要求理解这些新参数的工作原理，才能在异常场景下进行有效的故障排查和调优。
