## 一、药引子：为什么 Parquet 们不够用了？  
  
过去十年，Parquet 凭借出色的压缩率和分析性能，成为数据湖的事实标准。但在大模型时代，一个新痛点正在爆发： **随机访问（Random Access）性能**。  
  
想象你训练一个多模态 AI 模型，需要从数亿张图片中随机抽样 1 万张做小批次训练。如果用 Parquet，每次随机读一条记录，Parquet 必须从列文件末尾读取元数据、定位行组、再解压对应 Page——相比顺序扫描，性能可能下降 100 倍以上。  
  
这不是 Parquet 的缺陷——它面向的是**批量扫描分析**设计，随机读取从来不是它的目标。  
  
那么，AI 时代有没有一种列存格式，能同时兼顾：  
- 列存的压缩与向量化计算优势  
- 高速随机访问（随机读比 Parquet 快 100 倍）  
- 原生向量相似搜索  
- ACID 事务 + 时间旅行  
- 模式演进（Schema Evolution）而不重写全表  
  
这就是 Lance 试图解决的问题。  
  
## 二、反常识判断  
  
**真正值钱的不是"更好的 Parquet"，而是"随机读和批量扫描一样快"这个组合。**  
  
Parquet 的列存压缩比、编码技术业界领先，但它的设计前提是"数据懒加载、按需扫描"。一旦 AI 工作负载要求随机访问+向量化+实时写入这个组合，Parquet 的架构就成了瓶颈。  
  
Lance 的核心赌注： **用更细粒度的 MiniBlock（~4KB）替代 Parquet 的 Page（通常 8KB-1MB），让随机读一条记录只需解码极少数 MiniBlock，而批量扫描仍然保持接近 Parquet 的吞吐量。**  
  
## 三、Lance 是什么？  
  
Lance 是专为 AI 工作负载设计的开源列式存储格式，最初由 LanceDB 团队用 Rust 实现。它的核心定位：  
  
> An open lakehouse format designed for multimodal AI workloads.  
  
关键词： **开放**（不绑定云厂商）、**多模态**（向量+结构化+文本+图片）、**AI 原生**（随机读优先）。  
  
官方测试数据显示随机读性能比 Parquet/Iceberg **快 100 倍**，且不牺牲批量扫描性能。  
  
### 与 Arrow/Parquet 的关系  
  
| 维度 | Arrow | Parquet | Lance |  
|------|-------|---------|-------|  
| 设计目标 | 内存列式格式，跨进程零拷贝传输 | 磁盘列式存储，高压缩率批量分析 | 磁盘列式存储，高速随机访问+向量索引 |  
| 随机读 | 内存级最快 | 慢 | 极快（比 Parquet 快 100 倍） |  
| 压缩率 | 低（追求速度） | 高 | 中等 |  
| 向量索引 | 无 | 无 | IVF、HNSW、PQ/SQ/RQ |  
| ACID/时间旅行 | 无 | 依赖上层（Iceberg） | 原生支持 |  
| 生态集成 | Pandas、Polars、DuckDB、Spark | Hive、Spark、DuckDB、Presto | Arrow 生态（零拷贝）、LanceDB |  
  
简单说： **Arrow 面向内存，Parquet 面向磁盘批量扫描，Lance 面向磁盘随机访问+向量搜索。**  
  
## 四、存储层次结构  
  
Lance 的数据组织是一套清晰的层次：  
  
```  
Dataset（数据集）  
  └─ Manifest（清单，元数据快照）  
       └─ Fragment（水平分区，一批行）  
            ├─ DataFile（.lance 文件，存储部分列）  
            │    ├─ Page（数据页）  
            │    │    └─ MiniBlock（~4KB，编码/压缩单元）  
            │    └─ 文件结构：header + column pages + page table + footer  
            └─ Deletion File（软删除标记）  
```  
  
**核心设计要点：**  
  
- **Fragment** 是水平分区单元，每个 Fragment 包含部分行和部分列（不是所有列都存在每个 Fragment 里）。这减少了随机读的列投影成本。  
- **DataFile** 是 `.lance` 文件，包含文件头、列 pages、page table 和元数据页脚。  
- **MiniBlock** 是 ~4KB 的细粒度编码/压缩单元，比 Parquet 的 Page 小得多，因此随机读一条记录时只需解码极少数 MiniBlock。  
- **Fragment 可独立添加列**：新增列只需在每个 Fragment 末尾追加新 DataFile，不触发全表重写——这就是 Schema Evolution 的工程基础。  
  
## 五、编码与压缩  
  
Lance 在编码层面借鉴了列存的成熟技术，但有针对性取舍：  
  
| 编码方式 | 适用场景 | 说明 |  
|---------|---------|------|  
| Bitpacking | 定长整数 | 极简编码，每个值用最小 bit 数存储 |  
| Dictionary | 低基数字符串/整数 | 公共值存字典，DataPage 存序号 |  
| FSST | 字符串 | 类似于字典的字符串压缩，适合高重复字符串 |  
| RLE | 重复值序列 | 连续重复值用计数表示 |  
| 结构性编码 | 嵌套类型 | 使用 Repetition Level 和 Definition Level（与 Parquet/Dremel 同源）处理嵌套结构 |  
  
**设计哲学**：对比 Parquet 追求高压缩率，Lance 更倾向于"解压开销最小化"。因为 AI 场景中，随机读一条记录后往往立即传给 GPU/CPU 做向量化计算，解压延迟直接影响端到端 throughput。  
  
## 六、向量化搜索：原生向量索引  
  
Lance 的核心竞争力之一： **在列存格式内原生支持向量相似搜索索引**。  
  
向量索引的核心三要素，Lance 均有对应实现：  
  
### 1、聚类（Clustering）  
  
使用 **IVF（Inverted File Index）** ：通过 k-means 将向量空间划分成 N 个聚类（Cluster），搜索时只扫描相关聚类而非全量向量，大幅降低搜索计算量。  
  
### 2、子索引（Sub-Index）  
  
| 类型 | 说明 | 适用场景 |  
|------|------|---------|  
| FLAT | 精确搜索，无近似 | 小数据集或高精度需求 |  
| HNSW | 分层可导航小世界图，O(log N) 搜索 | 生产环境，高召回+低延迟 |  
  
### 3、量化（Quantization）  
  
| 类型 | 说明 | 压缩效果 |  
|------|------|---------|  
| PQ（Product Quantization） | 分段量化，每个子向量独立编码 | 高压缩，召回率适中 |  
| SQ（Scalar Quantization） | 标量量化，每维度独立 | 中等压缩，速度快 |  
| RQ（RabitQ） | 随机旋转+二元量化 | 极高压缩，适合极致内存受限场景 |  
  
常用索引组合：`IVF_PQ`、`IVF_HNSW_SQ`、`IVF_SQ`、`IVF_RQ`、`IVF_FLAT`。  
  
**索引存储**：向量索引本身作为普通 Lance 文件存储在 `_indices/` 目录，在 Manifest 中被引用。这使得索引和表数据天然一体化管理。  
  
## 七、ACID 事务与时间旅行  
  
Lance 通过 **MVCC（多版本并发控制）** 原生支持 ACID 事务和时间旅行：  
  
- 每个提交（Commit）原子性产生新的表版本，形成可串行化的历史。  
- Manifest 记录当前版本号和 Fragment 列表。  
- 任意历史版本可读取，实现零拷贝时间旅行。  
- 支持 Schema Evolution（如添加列），不触发全表重写，只在相关 Fragment 末尾追加。  
  
这比 Parquet+Iceberg 的方案更简洁——不需要独立的事务层，存储格式自身内嵌版本语义。  
  
## 八、LanceDB：建立在 Lance 上的无服务器向量数据库  
  
LanceDB 是基于 Lance 格式构建的向量数据库，核心特点：  
  
- **无服务器**：嵌入在应用程序中，无需独立进程，持久化存储直接用本地目录或对象存储（S3/Azure GCS）。  
- **多模态表**：一张表同时包含向量、文本、图片、元数据等异构字段。  
- **混合搜索**：向量相似搜索 + 全文搜索（BM25） + SQL 过滤可组合使用。  
- **多 SDK**：Python、TypeScript/JavaScript、Java、Rust。  
- **Arrow 生态集成**：零拷贝读取，可直接对接 Pandas、Polars、DuckDB、Spark、Ray。  
  
## 九、优势 & 适合场景  
  
**优势：**  
- 随机读性能比 Parquet 快 100 倍（官方 benchmark），同时保持接近 Parquet 的批量扫描吞吐量  
- 原生向量索引（IVF/HNSW/PQ/SQ/RQ），无需独立向量数据库即可支持 ANN 搜索  
- ACID 事务 + 时间旅行，版本管理原生内置  
- Schema Evolution 高效，添加列不重写全表  
- Rust 实现，性能好，内存安全  
- Arrow 生态零拷贝集成，Python/Java 绑定无序列化开销  
  
**适合场景：**  
- AI 训练数据管理（随机采样、增量写入）  
- 多模态 RAG（向量+文本+图片统一存储）  
- 向量数据库场景（替代 Pinecone/Qdrant 等纯向量库，获得结构化数据+事务能力）  
- 数据湖场景（需要比 Parquet 更好的随机读取性能）  
- 需要 Schema Evolution 的 ETL pipeline  
  
## 十、劣势 & 不适合场景  
  
**劣势：**  
- 压缩率不如 Parquet（因为更细粒度的 MiniBlock 和更轻量的压缩取舍）  
- 生态相比 Parquet 年轻，应用工具链还在成熟中  
- 没有像 Iceberg/Delta Lake 那样成熟的"表格式"治理体系（虽然 LanceDB 在补这方面）  
- 面向磁盘随机访问优化，在纯内存场景下 Arrow 仍是首选  
  
**不适合场景：**  
- 极致压缩率优先的冷存储场景（Parquet/OORC 更合适）  
- 已有成熟 Iceberg/Delta Lake 治理体系的数据湖（迁移成本高）  
- 小文件高频写入（MiniBlock 细粒度设计在极小数据量场景可能没有优势）

## 十二、快速上手  
  
### Python + LanceDB  
  
```python  
import lancedb  
from lancedb.pydantic import LanceModel, Vector  
  
class Item(LanceModel):  
    id: int  
    text: str  
    embedding: Vector(128)  
    price: float  
  
# 连接本地目录（自动创建）  
db = lancedb.connect("~/lancedb")  
  
# 创建表（schema 自动推断）  
table = db.create_table("items", schema=Item)  
  
# 写入数据  
table.add([  
    {"id": 1, "text": "apple", "embedding": [0.1] * 128, "price": 5.5},  
    {"id": 2, "text": "banana", "embedding": [0.2] * 128, "price": 3.0},  
])  
  
# 创建向量索引  
table.create_index(  
    vector_column_name="embedding",  
    index_type="IVF_HNSW_SQ",  # IVF聚类 + HNSW子索引 + SQ量化  
    num_partitions=256,  
    num_sub_vectors=96,  
)  
  
# 混合查询：向量相似 + SQL过滤  
results = table.search([0.1] * 128) \  
    .where("price < 4.0") \  
    .limit(5) \  
    .to_pandas()  
```  
  
### Rust 直接操作 Lance 格式  
  
```rust  
use lance::dataset::Dataset;  
  
let dataset = Dataset::open("s3://bucket/dataset").await?;  
  
// 随机读取单条记录（受益于 MiniBlock 细粒度设计）  
let batch = dataset.scan()  
    .with_row_id(42)  // 精确行号随机读  
    .execute()  
    .await?;  
```  
  
### DuckDB 使用 Lance 格式  
  
#### Install from DuckDB (recommended)  
  
If you just want to use the extension, install it directly from DuckDB's core extensions repository:  
  
```sql  
INSTALL lance;  
LOAD lance;  
  
SELECT *  
  FROM 'path/to/dataset.lance'  
  LIMIT 1;  
```  
  
See DuckDB's extension page for `lance` for the latest release details: https://duckdb.org/docs/stable/core_extensions/lance  
  
#### Build from source (development)  
  
This repository focuses on source builds for development and CI.  
  
1. Initialize submodules:  
  
```bash  
git submodule update --init --recursive  
```  
  
2. Build:  
  
```bash  
GEN=ninja make release  
```  
  
3. Load the extension from a standalone DuckDB binary (local builds typically require unsigned extensions):  
  
```bash  
duckdb -unsigned -c "LOAD 'build/release/extension/lance/lance.duckdb_extension'; SELECT 1;"  
```  
  
#### Query a Lance dataset  
  
```sql  
-- local file  
SELECT *  
  FROM 'path/to/dataset.lance'  
  LIMIT 10;  
-- s3  
SELECT *  
  FROM 's3://bucket/path/to/dataset.lance'  
  LIMIT 10;  
```  
  
To access object store URIs (e.g. `s3://...`), configure a `TYPE LANCE` secret .  
  
```sql  
CREATE SECRET (  
  TYPE LANCE,  
  PROVIDER credential_chain,  
  SCOPE 's3://bucket/'  
);  
  
SELECT *  
  FROM 's3://bucket/path/to/dataset.lance'  
  LIMIT 10;  
```  
  
#### Write a Lance dataset  
  
Use DuckDB's `COPY ... TO ...` to materialize query results as a Lance dataset.  
  
```sql  
-- Create/overwrite a Lance dataset from a query  
COPY (  
  SELECT 1::BIGINT AS id, 'a'::VARCHAR AS s  
  UNION ALL  
  SELECT 2::BIGINT AS id, 'b'::VARCHAR AS s  
) TO 'path/to/out.lance' (FORMAT lance, mode 'overwrite');  
  
-- Read it back via the replacement scan  
SELECT count(*) FROM 'path/to/out.lance';  
  
-- Append more rows to an existing dataset  
COPY (  
  SELECT 3::BIGINT AS id, 'c'::VARCHAR AS s  
) TO 'path/to/out.lance' (FORMAT lance, mode 'append');  
  
-- Optionally create an empty dataset (schema only)  
COPY (  
  SELECT 1::BIGINT AS id, 'x'::VARCHAR AS s  
  LIMIT 0  
) TO 'path/to/empty.lance' (FORMAT lance, mode 'overwrite', write_empty_file true);  
```  
  
To write to `s3://...` paths, configure a `TYPE LANCE` secret for that scope .  
  
```sql  
CREATE SECRET (  
  TYPE LANCE,  
  PROVIDER credential_chain,  
  SCOPE 's3://bucket/'  
);  
  
COPY (SELECT 1 AS id) TO 's3://bucket/path/to/out.lance' (FORMAT lance, mode 'overwrite');  
```  
  
#### Create a Lance dataset via `CREATE TABLE` (directory namespace)  
  
When you `ATTACH` a directory as a Lance namespace, you can create new datasets using `CREATE TABLE` (schema-only)  
or `CREATE TABLE AS SELECT` (CTAS). The dataset is written to `<namespace_root>/<table_name>.lance`.  
  
```sql  
ATTACH 'path/to/dir' AS lance_ns (TYPE LANCE);  
  
-- Schema-only (creates an empty dataset)  
CREATE TABLE lance_ns.main.my_empty (id BIGINT, s VARCHAR);  
  
-- CTAS (writes query results)  
CREATE TABLE lance_ns.main.my_dataset AS  
  SELECT 1::BIGINT AS id, 'a'::VARCHAR AS s  
  UNION ALL  
  SELECT 2::BIGINT AS id, 'b'::VARCHAR AS s;  
  
SELECT count(*) FROM lance_ns.main.my_dataset;  
```  
  
#### Vector search  
  
```sql  
-- Search a vector column, returning distances in `_distance` (smaller is closer)  
SELECT id, label, _distance  
FROM lance_vector_search('path/to/dataset.lance', 'vec', [0.1, 0.2, 0.3, 0.4]::FLOAT[4],  
                         k = 5, prefilter = true)  
ORDER BY _distance ASC;  
```  
  
  
#### Full-text search (FTS)  
  
```sql  
-- Search a text column, returning BM25-like scores in `_score`  
SELECT id, text, _score  
FROM lance_fts('path/to/dataset.lance', 'text', 'puppy', k = 10, prefilter = true)  
ORDER BY _score DESC;  
```  
  
  
#### Hybrid search (vector + FTS)  
  
```sql  
-- Combine vector and text scores, returning `_hybrid_score` in addition to `_distance` / `_score`  
SELECT id, _hybrid_score, _distance, _score  
FROM lance_hybrid_search('path/to/dataset.lance',  
                         'vec', [0.1, 0.2, 0.3, 0.4]::FLOAT[4],  
                         'text', 'puppy',  
                         k = 10, prefilter = false,  
                         alpha = 0.5, oversample_factor = 4)  
ORDER BY _hybrid_score DESC;  
```  
  
## 十三、基准测试参考  
  
官方 benchmark（Github: lancedb/lance）显示：  
  
- **随机读**：比 Parquet 快 **100 倍**（1M 行数据集，随机采样 1% 行）  
- **批量扫描**：性能与 Parquet 持平（差距 < 20%）  
- **向量搜索延迟**：1M 768 维向量，IVF_PQ 索引，QPS > 3000（单节点）  
  
核心原理：MiniBlock ~4KB 粒度 + 列投影裁剪 + 无需解压完整 Page 即定位目标行。  
  
## 十四、边界条件  
  
**这个结论需要以下前提：**  
  
1. 随机读取比例不高但 latency 要求严（AI 训练小批次采样）  
2. 需要向量搜索+结构化数据混合查询  
3. 需要 Schema Evolution 但不想做全量重写  
4. 可接受略低于 Parquet 的压缩率  
  
**如果前提崩塌：**  
  
- 如果你追求极致压缩率 + 全表批量扫描为主 → Parquet/Iceberg 仍是首选  
- 如果你已有成熟数据湖治理体系 → Lance 迁移成本可能划不来  
- 如果纯内存分析为主 → Arrow 是更自然的选择  
  
## 十五、扩展问题  
  
### 1、Lance 的 MiniBlock 粒度设计 vs Parquet Page，核心差异在哪？  
  
Parquet 的 Page 通常 8KB-1MB，随机读一条记录可能需要解压整个 Page。Lance 的 MiniBlock ~4KB，随机读只需解码目标行所在的极少数 MiniBlock。代价是元数据开销略高（每个 MiniBlock 需要独立元信息），因此 Lance 选择用中低压缩率换随机读性能。  
  
### 2、Lance 的向量索引和 pgvector 的 HNSW 有什么区别？  
  
pgvector 的 HNSW 是 PostgreSQL 内置索引，随 PG 表一起管理，适合与结构化 SQL 联表查询。Lance 的向量索引是独立文件，与表数据同一格式管理，适合纯向量工作负载或非 PG 环境。两者都支持 IVF+HNSW+量化组合，核心算法层面无本质差异。  
  
### 3、Lance vs Delta Lake / Iceberg？  
  
Delta Lake 和 Iceberg 是**表格式**（Table Format），定义的是"如何组织一群 Parquet/ORC 文件使其成为一张可版本化、可事务操作的表"。Lance 是**存储格式**本身，同时内嵌了版本语义。两者不在同一层级——可以理解为：Iceberg 管理 Parquet 文件，Lance 自己就是文件格式+版本管理一体。
