## 背景  

本文深入解读 PostgreSQL 核心代码中关于查询级 Instrumentation 机制的这次重要重构（commit 2c16deee2f7d52d6567dcbad046f74a8e880ee52）。该变更将查询级别统计信息的分配责任从外部扩展移交至核心执行器，通过引入 `query_instr_options` 选项字段，实现了多扩展间的可组合协作，解决了长期困扰扩展开发者的"选项覆盖"痛点。

  

## 背景痛点

在 PostgreSQL 生态中，`pg_stat_statements` 和 `auto_explain` 是两大必备扩展。前者负责收集查询统计信息，后者自动记录慢查询的执行计划。它们都需要追踪"整个查询的执行时间"这一指标。

### 旧架构的问题

在旧的设计中，扩展需要**自己动手**分配和初始化查询级 Instrumentation 结构体：

```c
/* auto_explain.c 旧代码 */
if (queryDesc->totaltime == NULL)
{
    MemoryContext oldcxt;
    oldcxt = MemoryContextSwitchTo(queryDesc->estate->es_query_cxt);
    queryDesc->totaltime = InstrAlloc(INSTRUMENT_ALL);  // 扩展自行分配！
    MemoryContextSwitchTo(oldcxt);
}
```

这种设计存在三个核心问题：

1. **可组合性差**：如果扩展 A 设置了 `INSTRUMENT_TIMER`，扩展 B 设置了 `INSTRUMENT_BUFFERS`，两者会相互覆盖，最终只有一个生效
2. **强制全开**：为了确保另一个扩展也能正常工作，所有扩展都不得不使用 `INSTRUMENT_ALL`，造成不必要的性能开销
3. **语义误导**：`totaltime` 这个名字暗示"只测时间"，但实际上 Instrumention 框架早已支持 buffers、WAL、IO 等多种资源追踪

## 原理介绍

### 核心数据结构

理解这个改动，首先要理解 `QueryDesc` 结构体中与 Instrumentation 相关的字段：

```c
typedef struct QueryDesc
{
    /* ... 其他字段 ... */

    /* 节点级instrumentation选项（每个执行节点） */
    int     instrument_options;          /* OR of InstrumentOption flags */

    /* 新增：查询级instrumentation选项（由扩展设置） */
    int     query_instr_options;         /* OR of InstrumentOption flags */

    /* 查询级instrumentation结构体（由ExecutorStart分配） */
    struct Instrumentation *query_instr; /* query level instrumentation */
} QueryDesc;
```

`InstrumentOption` 定义了支持的统计类型：

```c
typedef enum InstrumentOption
{
    INSTRUMENT_TIMER   = 1 << 0,  /* 计时器 + 行数 */
    INSTRUMENT_BUFFERS = 1 << 1,  /* 缓冲区使用 */
    INSTRUMENT_ROWS    = 1 << 2,  /* 行计数 */
    INSTRUMENT_WAL     = 1 << 3,  /* WAL 使用 */
    INSTRUMENT_IO      = 1 << 4,  /* IO 使用 */
    INSTRUMENT_ALL     = PG_INT32_MAX  /* 全部开启 */
} InstrumentOption;
```

### 新架构的执行流程

**1. CreateQueryDesc（查询描述符创建）**

```c
/* src/backend/tcop/pquery.c */
qd->query_instr_options = 0;    /* 初始化为0，表示暂无扩展需要 */
qd->query_instr = NULL;         /* 标记为未分配 */
```

**2. 扩展通过 Hook 设置需求**

扩展在 `ExecutorStart` Hook 中通过 `|=` 操作符**添加**自己需要的选项：

```c
/* pg_stat_statements.c */
pgss_ExecutorStart(QueryDesc *queryDesc, int eflags)
{
    if (pgss_enabled(nesting_level) && queryDesc->plannedstmt->queryId != INT64CONST(0))
    {
        /* 请求所有统计信息：时间 + 缓冲区 + WAL */
        queryDesc->query_instr_options |= INSTRUMENT_ALL;
    }

    /* 调用下一个Hook或标准ExecutorStart */
    if (prev_ExecutorStart)
        prev_ExecutorStart(queryDesc, eflags);
    else
        standard_ExecutorStart(queryDesc, eflags);
}
```

```c
/* auto_explain.c */
explain_ExecutorStart(QueryDesc *queryDesc, int eflags)
{
    if (auto_explain_enabled())
    {
        /* 只需要计时器，不需要buffers和wal */
        queryDesc->query_instr_options |= INSTRUMENT_TIMER;
        /* ... 其他设置 ... */
    }
    /* ... */
}
```

**关键点**：使用 `|=` 而不是 `=`，确保多个扩展的选项是**叠加**关系，而非覆盖关系。

**3. ExecutorStart 统一分配**

核心执行器在 `standard_ExecutorStart` 中根据所有扩展的需求，统一分配 Instrumentation 结构：

```c
/* src/backend/executor/execMain.c */
standard_ExecutorStart(QueryDesc *queryDesc, int eflags)
{
    /* ... estate 初始化 ... */

    /*
     * 如果有扩展请求了查询级instrumentation，则统一分配
     * 注意：Assert确保扩展不能再自行分配
     */
    Assert(queryDesc->query_instr == NULL);
    if (queryDesc->query_instr_options)
        queryDesc->query_instr = InstrAlloc(queryDesc->query_instr_options);

    /* ... 后续初始化 ... */
}
```

**4. 执行阶段自动计时**

```c
standard_ExecutorRun(QueryDesc *queryDesc, ...)
{
    /* Allow instrumentation of Executor overall runtime */
    if (queryDesc->query_instr)
        InstrStart(queryDesc->query_instr);  /* 开始计时 */

    /* ... 执行查询 ... */

    if (queryDesc->query_instr)
        InstrStop(queryDesc->query_instr);   /* 停止计时 */
}
```

### 字段重命名语义

| 旧字段名 | 新字段名 | 说明 |
|---------|---------|------|
| `totaltime` | `query_instr` | 名字更准确：它是"查询级统计信息"，不只有时间 |
| — | `query_instr_options` | 新增：让扩展声明需要哪些统计指标 |

## 应用场景及最佳实践

### 场景一：pg_stat_statements + auto_explain 共存

**场景描述**：DBA 同时启用了 `pg_stat_statements` 和 `auto_explain` 两个扩展。

**修改前的问题**：

假设 `auto_explain` 先设置 `INSTRUMENT_TIMER`，`pg_stat_statements` 后设置 `INSTRUMENT_ALL`。如果实现不当，两者会相互覆盖。

**修改后的行为**：

```
pg_stat_statements 设置: query_instr_options |= INSTRUMENT_ALL   (时间+缓冲+WAL)
auto_explain 设置:      query_instr_options |= INSTRUMENT_TIMER  (只需时间)
                         ↓
合并结果:                query_instr_options = INSTRUMENT_ALL | INSTRUMENT_TIMER
                         ↓
ExecutorStart 分配:      InstrAlloc(INSTRUMENT_ALL | INSTRUMENT_TIMER)
                         ↓
两个扩展都获得了所需的全部信息！
```

### 场景二：auto_explain 精确请求所需选项

**修改前**（auto_explain 旧代码）：

```c
/* 强制请求所有instrumentation，因为担心其他扩展需要 */
queryDesc->totaltime = InstrAlloc(INSTRUMENT_ALL);
```

**修改后**（auto_explain 新代码）：

```c
/* 只请求计时器，auto_explain只需要总执行时间 */
queryDesc->query_instr_options |= INSTRUMENT_TIMER;

/* 如果需要 ANALYZE，才会请求 per-node 的详细统计 */
if (auto_explain_log_analyze && (eflags & EXEC_FLAG_EXPLAIN_ONLY) == 0)
{
    /* per-node instrumentation 通过 estate->es_instrument 设置 */
}
```

**优化效果**：减少了不必要的数据收集，提升性能。

### 场景三：扩展开发者的新范式

**旧范式（扩展自己管理分配）**：

```c
/* 扩展代码 */
void my_extension_ExecutorStart(QueryDesc *queryDesc, int eflags)
{
    /* 调用下层 */
    prev_ExecutorStart(queryDesc, eflags);

    /* 自己分配！这是不推荐的做法 */
    if (need_instrumentation)
        queryDesc->totaltime = InstrAlloc(INSTRUMENT_ALL);
}
```

**新范式（声明式选项）**：

```c
/* 扩展代码 */
void my_extension_ExecutorStart(QueryDesc *queryDesc, int eflags)
{
    /* 先声明需求 */
    if (need_instrumentation)
        queryDesc->query_instr_options |= INSTRUMENT_TIMER;  /* 或 INSTRUMENT_ALL */

    /* 调用下层，分配由核心负责 */
    if (prev_ExecutorStart)
        prev_ExecutorStart(queryDesc, eflags);
    else
        standard_ExecutorStart(queryDesc, eflags);
}
```

## 小结和思考

### 核心价值

1. **可组合性**：多个扩展可以安全地叠加各自的 instrumentation 需求，不再相互覆盖
2. **职责清晰**：分配职责归位到核心执行器，扩展只需声明需求（声明式而非命令式）
3. **语义准确**：`query_instr` 这个名字直观表明其用途，避免 `totaltime` 带来的误解
4. **性能优化**：扩展可以精确请求所需的统计类型，避免 `INSTRUMENT_ALL` 的过度开销

### 对 DBA 的实际意义

- **更稳定的扩展组合**：同时启用多个需要查询级统计的扩展时，不再出现"互相打架"的情况
- **更少的性能开销**：每个扩展只请求自己真正需要的统计选项
- **更清晰的行为预期**：扩展的行为变得更加可预测和透明

### 对扩展开发者的建议

1. **使用 `|=` 而非 `=`**：设置 `query_instr_options` 时务必使用 `|=`，保留其他扩展已设置的选项
2. **精确请求**：只请求你真正需要的选项（`INSTRUMENT_TIMER` 而非 `INSTRUMENT_ALL`）
3. **不要自行分配**：不要尝试自己调用 `InstrAlloc` 分配 `query_instr`，这会触发 Assert 失败

### 架构演进思考

这个改动体现了 PostgreSQL 架构设计的一个重要趋势：**从"你来分配我来调用"到"你来声明我来分配"**。这种模式在现代系统设计中广泛存在（如 Kubernetes 的声明式 API），能够更好地支持多组件共存的可组合系统。
