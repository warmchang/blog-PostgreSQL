## 背景 

`pg_stash_advice` 是 PostgreSQL 17 新增的 contrib 模块（commit e8ec19aa321abc89fb4fd277c994f14680ba17db，作者 Robert Haas），它将 `pg_plan_advice` 生成的执行计划建议以"命名仓库"（advice stash）的形式存储在动态共享内存（DSA/D中，通过查询标识符（queryId）实现自动命中，让经过验证的优化建议在生产环境中持续生效，而无需每次手动注入。

核心关键词：`pg_stash_advice , pg_plan_advice , queryId , Dynamic Shared Memory , dshash , advice stash , 计划建议持久化 , GUC 会话级配置 , LWLock 并发控制 , advisor hook`

背景痛点：DBA 在使用 `pg_plan_advice` 时，建议只能通过 `PLAN_ADVICE` 选项在单次 `EXPLAIN` 中被动产生，无法持久化到后续运行中。每次执行同一个 SQL 都要重新运行 `pg_plan_advice` 才能获得建议。生产环境中，如果某个慢查询因规划器统计信息过期或 bug 导致选错执行计划，DBA 往往只能通过 `hint` 补丁或修改应用代码来干预，极不灵活，且无法跨会话共享。

本模块的价值与意义：它将"一次性"的计划建议变成了一种可管理、可切换、可按查询精确命中的**运行时控制面**——DBA 可以在一个会话中分析查询、将验证过的建议存入 stash，再让其他会话自动受益，真正实现了"分析一次，持续生效"的工作流。

## 原理介绍

### 1. 模块架构总览

`pg_stash_advice` 由两个核心源文件和一个 SQL 接口文件组成：

| 文件 | 职责 |
|------|------|
| `pg_stash_advice.c` | 核心引擎：内存分配、共享状态初始化、advisor hook 实现 |
| `stashfuncs.c` | SQL 接口层：5 个 SQL 函数的具体实现 |
| `pg_stash_advice--1.0.sql` | DDL 脚本：创建 5 个 SQL 函数，回收 PUBLIC 执行权限 |

模块通过 `pg_plan_advice` 提供的 `advisor hook` 机制接入 PostgreSQL 的查询规划流程，无需修改核心代码。

### 2. 初始化阶段：`_PG_init` 做了三件事

在 `pg_stash_advice.c` 的 `_PG_init()` 函数（第 631-657 行）中，模块加载时完成三项关键注册：

```c
void
_PG_init(void)
{
    void (*add_advisor_fn) (pg_plan_advice_advisor_hook hook);

    /* 第一件事：确保 queryId 被计算（如果 GUC=auto 则自动启用） */
    EnableQueryId();

    /* 第二件事：定义会话级 GUC，默认为空字符串（禁用状态） */
    DefineCustomStringVariable("pg_stash_advice.stash_name",
                               "Name of the advice stash to be used in this session.",
                               NULL,
                               &pg_stash_advice_stash_name,
                               "",                          /* 默认值：空 = 禁用 */
                               PGC_USERSET,                /* 会话级可调 */
                               0,
                               pgsa_check_stash_name_guc, /* 校验 hook */
                               NULL, NULL);
    MarkGUCPrefixReserved("pg_stash_advice");

    /* 第三件事：将自身注册为 pg_plan_advice 的 advisor */
    add_advisor_fn =
        load_external_function("pg_plan_advice", "pg_plan_advice_add_advisor",
                               true, NULL);
    (*add_advisor_fn) (pgsa_advisor);
}
```

**源码解读要点：**

- `EnableQueryId()`（定义于 `src/include/nodes/queryjumble.h:95`）确保即使 `compute_query_id = auto`，也会启用 queryId 计算。整个 stash 机制依赖 queryId 作为查找键，没有 queryId 就没有命中。
- `pg_stash_advice.stash_name` 属于 `PGC_USERSET`，即普通会话用户可直接 `SET`，这意味着不同会话可以同时绑定不同的 stash，互不干扰。
- `load_external_function` 通过动态加载 `pg_plan_advice` 的符号表，将其作为一个"插件框架"使用——这是 PostgreSQL 扩展间解耦通信的经典模式。

### 3. advisor hook：规划阶段自动查找建议

当 PostgreSQL 规划一个查询时，`pg_plan_advice` 的规划逻辑会依次调用已注册的 advisor hook。`pg_stash_advice` 通过 `pgsa_advisor()` 提供建议字符串（`pg_stash_advice.c` 第 663-720 行）：

```c
static char *
pgsa_advisor(PlannerGlobal *glob, Query *parse,
             const char *query_string, int cursorOptions,
             ExplainState *es)
{
    pgsa_entry_key key;
    pgsa_entry *entry;
    char *advice_string;
    uint64 stash_id;

    /* 快速短路：stash 未配置或无 queryId 则直接返回 NULL */
    if (pg_stash_advice_stash_name[0] == '\0' || parse->queryId == 0)
        return NULL;

    /* 按需附着动态共享内存 */
    if (unlikely(pgsa_entry_dshash == NULL))
        pgsa_attach();

    /* 将 stash 名称转换为整数 ID */
    stash_id = pgsa_lookup_stash_id(pg_stash_advice_stash_name);
    if (stash_id == 0)
        return NULL;

    /* 构造复合查找键：(stash_id, queryId) */
    memset(&key, 0, sizeof(pgsa_entry_key));
    key.pgsa_stash_id = stash_id;
    key.queryId = parse->queryId;

    entry = dshash_find(pgsa_entry_dshash, &key, false);
    if (entry == NULL)
        return NULL;

    /* 将 DSA 指针转换为实际字符串，并复制到会话内存上下文中 */
    if (entry->advice_string == InvalidDsaPointer)
        advice_string = NULL;
    else
        advice_string = pstrdup(dsa_get_address(pgsa_dsa_area,
                                                entry->advice_string));
    dshash_release_lock(pgsa_entry_dshash, entry);

    if (advice_string != NULL)
        elog(DEBUG2, "supplying automatic advice for stash \"%s\", query ID %" PRId64 ": %s",
             pg_stash_advice.stash_name, key.queryId, advice_string);

    return advice_string;
}
```

**源码解读要点：**

- 复合键 `(stash_id, queryId)` 是整个查找机制的核心——一个 stash 中可以存储多个不同查询的建议，通过 queryId 精确区分。
- `dsa_get_address` 将 DSA 区域中的指针转换为本地可访问的地址，`pstrdup` 将其复制到当前内存上下文（通常是会话级短期上下文），之后立即释放 dshash 锁，保证最小化锁持有时间。
- `pg_plan_advice` 中按 hook 注册顺序调用 advisors，第一个返回非 NULL 字符串的即被使用（`pg_plan_advice.c` 第 177-189 行）。`pg_stash_advice` 注册在前（用户显式加载），`pg_plan_advice` 内置建议在后。

### 4. 共享内存结构：三层数据结构

`pg_stash_advice` 的共享内存布局由 `pgsa_shared_state` 统一管理（`pg_stash_advice.h` 第 1226-1236 行）：

```c
typedef struct pgsa_shared_state
{
    LWLock      lock;              /* 统一的互斥锁 */
    int         dsa_tranche;       /* DSA 区域的 tranche ID */
    int         stash_tranche;     /* stash 名称哈希表的 tranche ID */
    int         entry_tranche;     /* entry 哈希表的 tranche ID */
    uint64      next_stash_id;     /* 自增 stash ID 分配器 */
    dsa_handle  area;              /* DSA 区域句柄（重启后可重新 attach） */
    dshash_table_handle stash_hash; /* stash_name -> pgsa_stash 哈希表句柄 */
    dshash_table_handle entry_hash; /* (stash_id, queryId) -> pgsa_entry 哈希表句柄 */
} pgsa_shared_state;
```

共享内存通过 `GetNamedDSMSegment` 创建命名段（`pg_stash_advice.c` 第 198-202 行），会话启动后首次使用时通过 `dsa_attach` 重新附着，不需要重启服务器。`next_stash_id` 从 1 开始自增，保证每个 stash 拥有唯一的整数 ID，stash 名称到 ID 的映射存储在 `pgsa_stash_dshash` 哈希表中。

两种哈希表的键类型不同，体现了设计上的精心选择：

- `pgsa_stash_dshash`：键类型为 `char *`（变长字符串），使用 `dshash_strcmp/dshash_strhash`——因为查找场景是按 stash 名称精确匹配。
- `pgsa_entry_dshash`：键类型为 `pgsa_entry_key`（固定长度结构体 `{uint64, int64}`），使用 `dshash_memcmp/dshash_memhash`——因为查找场景是复合键精确匹配。

### 5. SQL 接口函数

所有 SQL 函数在 `stashfuncs.c` 中实现，核心操作均通过 `LWLock` 保护并发：

| SQL 函数 | 底层操作 | 锁模式 |
|----------|----------|--------|
| `pg_create_advice_stash(name)` | 调用 `dshash_find_or_insert` 插入 stash 记录，分配 stash_id | `LW_EXCLUSIVE` |
| `pg_drop_advice_stash(name)` | 删除 stash 记录 + 遍历删除所有关联 entry | `LW_EXCLUSIVE` |
| `pg_set_stashed_advice(name, queryId, advice)` | 在 entry 哈希表中插入/更新 advice 字符串 | `LW_SHARED`（读取）+ 调用方持有锁 |
| `pg_get_advice_stashes()` | 遍历 stash 哈希表，聚合计数 | 无锁（只读遍历） |
| `pg_get_advice_stash_contents(name)` | 遍历 entry 哈希表，按 stash_id 过滤 | 无锁（只读遍历） |

值得注意的是 `pg_set_stashed_advice` 传入 `advice_string = NULL` 时调用 `pgsa_clear_advice_string`（`pg_stash_advice.c` 第 391-426 行），该函数先删除 entry，再释放 `dsa_free` 分配的 advice 字符串内存，防止内存泄漏。

### 6. stash 命名校验：跨数据库安全设计

`pgsa_check_stash_name` 函数（`pg_stash_advice.c` 第 837-870 行）对 stash 名称施加三重校验：

1. **非空**：stash 名称不能为空字符串（空 = 禁用功能）
2. **长度限制**：不超过 `NAMEDATALEN - 1 = 63` 字节（PostgreSQL 标准标识符长度）
3. **纯 ASCII + 标识符格式**：必须以字母或下划线开头，后续字符只能是字母、数字、下划线

第三点特别重要——因为 advice stash 存储在动态共享内存中，跨所有数据库可见。如果允许非 ASCII 名称，不同数据库编码不同会导致名称解释不一致。因此 stash 名称被设计为"数据库无关的全局标识符"。

## 应用场景及最佳实践

### 场景一：统计信息过期导致索引被忽略的紧急干预

**背景**：某业务表 `orders` 在 `status` 列上新建立了索引，但 PostgreSQL 的统计信息因为 `autovacuum` 未及时触发，导致规划器估算行数严重偏差（估计 1 行，实际 100 万行），选择了 Seq Scan 而非 Index Scan。

**干预前（DBA 手动方式）**：
```sql
-- DBA 每次发现慢查询，都需要：
EXPLAIN (PLAN_ADVICE ON) SELECT * FROM orders WHERE status = 'pending';
-- 看到 pg_plan_advice 给出的 INDEX_SCAN 建议
-- 然后通知应用团队在 SQL 中加 hint，或改写 SQL
-- 下次遇到同样的查询，又得重来一遍
```

**干预后（pg_stash_advice 方式）**：
```sql
-- 第一步：DBA 在管理会话中获取 queryId 并存入 stash
CREATE EXTENSION pg_stash_advice;
SET compute_query_id = on;
SELECT pg_create_advice_stash('urgent_fix');

-- 获取慢查询的 queryId（从 pg_stat_activity 或 EXPLAIN VERBOSE）
SELECT pg_set_stashed_advice('urgent_fix', :slow_query_id,
    'INDEX_SCAN(orders orders_status_idx)');

-- 第二步：在受影响的应用会话中启用
SET pg_stash_advice.stash_name = 'urgent_fix';
-- 之后该会话中所有命中 queryId 的查询自动应用 INDEX_SCAN

-- 第三步：等 autovacuum 跑完、统计信息更新后
SELECT pg_set_stashed_advice('urgent_fix', :slow_query_id, NULL);
-- 清除 stash 中的建议，规划器恢复正常决策
```

**对比**：
- 前：每次都要手动干预，无法跨会话共享，效率低且容易遗漏
- 后：分析一次，存入 stash，立即生效，清除后自动恢复——完整闭环

### 场景二：按角色/数据库隔离不同优化策略

**背景**：某 DBA 团队想在一套测试环境中同时验证两种不同的 join 顺序策略，但不想互相干扰。

**干预前**：
- 策略 A 的建议写在数据库级别的配置中
- 策略 B 需要改动应用代码或用不同用户连接
- 无法做到"同一实例中同时生效两种不同建议"

**干预后**：
```sql
-- DBA 创建两个独立的 stash
SELECT pg_create_advice_stash('strategy_a');
SELECT pg_create_advice_stash('strategy_b');

-- 向 strategy_a 存入 join_order 建议
SELECT pg_set_stashed_advice('strategy_a', :qid, 'join_order(t1 t2 t3)');

-- 向 strategy_b 存入 NESTED_LOOP 建议
SELECT pg_set_stashed_advice('strategy_b', :qid, 'NESTED_LOOP(t1 t2)');

-- 会话 1：使用策略 A
SET pg_stash_advice.stash_name = 'strategy_a';

-- 会话 2：使用策略 B
SET pg_stash_advice.stash_name = 'strategy_b';

-- 会话 3：不使用任何 stash（普通规划）
SET pg_stash_advice.stash_name = '';
```

**对比**：三个会话同时连接同一数据库，对同一个 SQL 产生三种不同的执行计划，各自独立、互不干扰。这对于 A/B 测试执行计划效果极为有用。

### 场景三：多阶段发布中的计划干预回滚

**背景**：某系统升级 PostgreSQL 大版本后，某个复杂 OLAP 查询的规划行为发生变化，DBA 想在过渡期用 stash 强制稳定计划，新版本稳定后逐步退出干预。

```sql
-- 发布前：创建干预 stash
SELECT pg_create_advice_stash('v15_transition');

-- 逐个存入受影响查询的建议
SELECT pg_set_stashed_advice('v15_transition', :qid1, 'SET enable_hashjoin TO off');
SELECT pg_set_stashed_advice('v15_transition', :qid2, 'SET enable_seqscan TO off');

-- 发布时：应用会话启用 stash
SET pg_stash_advice.stash_name = 'v15_transition';

-- 发布稳定 2 周后：按查询逐个清除 stash 记录
SELECT pg_set_stashed_advice('v15_transition', :qid1, NULL);  -- 先清除一个

-- 确认性能正常后，清除全部
SELECT pg_drop_advice_stash('v15_transition');
```

### 最佳实践总结

1. **将 stash 加载脚本化**：由于 stash 存储在内存中，服务器重启后需要重新创建和填充。建议将 `pg_create_advice_stash` 和 `pg_set_stashed_advice` 调用写入 SQL 脚本，配合 `shared_preload_libraries` 自动化加载。
2. **用 `ALTER DATABASE/ROLE SET` 精确控制生效范围**：将 `pg_stash_advice.stash_name` 通过数据库级或角色级 GUC 设置，避免全局生效影响无辜查询。
3. **优先使用最小化建议字符串**：Advice 字符串只需包含真正需要干预的部分（如仅 `INDEX_SCAN` 而非整条计划），可减少解析开销和出错概率。
4. **定期清理无用的 stash 条目**：当查询被废弃或应用逻辑变更后，及时用 `pg_set_stashed_advice(..., NULL)` 清除对应条目，避免共享内存持续膨胀。
5. **理解安全边界**：`pg_stash_advice` 目前缺乏细粒度安全模型，任何会话用户都能通过 `EXPLAIN VERBOSE` 看到查询的 queryId。建议仅将已验证过的建议放入 stash，避免暴露业务敏感的查询特征。

## 小结和思考

`pg_stash_advice` 的核心创新在于**将 `pg_plan_advice` 从一个"一次性分析工具"转变为一个"可持续生效的优化建议引擎"**。它利用了 PostgreSQL 的三个既有基础设施：

- `pg_plan_advice` 的 advisor hook 插件架构
- 动态共享内存（DSA/DSM）的跨进程共享能力
- `compute_query_id` 提供的查询指纹机制

这三条线的交汇，产生了一个简洁而强大的机制：queryId 作为"键"，advice 字符串作为"值"，stash 作为"命名空间隔离层"。

对 DBA 而言，这意味着计划干预从 **"临时补丁"** 演进为 **"可管理的运行时配置"** 。DBA 可以建立自己的"优化建议知识库"：经过 `pg_plan_advice` 分析并验证有效的建议，存入 stash 后持续生效；发现问题后可以随时清除，无需改代码、不用重启服务。

同时也要清醒认识到它的边界：stash 是运行时的内存设施，不是持久化配置；它不能替代统计信息更新和正确的表设计——这些才是解决规划器错误的根本手段。`pg_stash_advice` 是"知道哪里有坑，先绕过去"的战术工具，而统计信息和表结构优化才是"把坑填平"的战略投资。
