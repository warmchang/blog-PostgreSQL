## 背景
最近 PostgreSQL 生态里出了一个有意思的开源扩展 —— pg_turbovec。

它做的事情一句话讲清楚: 把 Google Research 在 ICLR 2026 上发表的 TurboQuant 向量量化算法,包装成一个 Postgres 原生的向量索引访问方法 (Index Access Method), 底层依赖 Ryan Codrai 写的 turbovec Rust crate。

具体能省多少?在 100 万条 1536 维的真实 OpenAI 嵌入上, 4-bit 模式单向量实测 780 字节; 比 pgvector 那个未压缩的 HNSW 版本, **整整 10~20 倍的存储差距**。 

但如果我告诉你,这个省 20 倍内存的方案,有一种用法比 pgvector HNSW 慢 **大约 490 倍** —— 你还会用吗? 

先别急着下结论。pg_turbovec 是一个「只有理解了它的适用边界, 才能用对」的项目。今天这篇文章就把它的核心数字、底层原理、3 种索引类型、以及最重要的**老实边界**讲清楚。

先别翻走, 它有索引选项, 速度贼快.  

  

## 一、被低估的「向量内存压力」

在讨论 pg_turbovec 之前,先看一个绝大多数 AI 工程师都低估的问题: **向量数据库的内存成本**。

假设你的 RAG 系统用了 pgvector,塞了 100 万条 1536 维 OpenAI 嵌入 (这个量级对中等规模企业 RAG 来说是起步价):

- 一条 FP32 向量 = 6144 字节 (1536 × 4 字节)
- 100 万条 = **6 GB**
- 1000 万条 = **60 GB**
- 1 亿条 = **600 GB**

HNSW 索引还要额外存储图结构,实际内存占用通常是向量本身的 1.3~2 倍。

绝大部分生产环境的内存预算根本撑不到这种规模。所以现实里大家被迫做选择:

- 要么放弃 HNSW,接受暴力扫描 (慢得没法看)
- 要么用 pgvector 自带的 `binary_quantize()` + `bit_hamming_ops` HNSW (32× 压缩,但 Recall@10 直接掉到 0.65~0.75)
- 要么彻底离开 Postgres,迁移到 Milvus、Qdrant、Weaviate 之类的专用向量数据库

pg_turbovec 想做的事,是在前两个选项之间找到一个**更聪明的中间点** —— 在 Postgres 内部,提供一种压缩比与 Hamming 相当、但召回率显著更高的量化方案。

  

## 二、TurboQuant 的「码本无关」魔法

理解 pg_turbovec 的关键,是理解它底层用的 TurboQuant 算法。

它做了一件看似违反直觉的事: **完全不看具体数据**,直接给所有 L2 归一化向量做 5 步处理:

1. **L2 归一化**:把向量的长度剥掉,只保留方向。
2. **随机正交旋转**:用一个固定 (与数据无关) 的随机正交矩阵把向量转个方向。
3. **Lloyd-Max 标量量化**:对每个坐标独立做 k-means,得到最优的量化边界。
4. **位打包**:把多个坐标打包成 2/3/4 比特/坐标 的位编码。
5. **长度重归一化打分**:打回近似内积时修正一个尺度因子。

为什么这种「不看数据」的方案反而能赢?

数学上有一个反直觉的事实: **L2 归一化的高维向量经过随机正交矩阵旋转后,每个坐标独立收敛到一个均值 0、方差 1/d 的正态分布**。这件事是数学推导出来的,跟你具体是什么语料、什么嵌入模型无关。

既然理论分布已知,那么「给这个分布找最优量化边界」就被简化成了一个标准的 1D k-means 问题,**用 Lloyd-Max 算法提前算好、存好就行**。码本完全由数学公式决定,与具体数据无关。

这件事直接带来三个工程上的好处:

1. **不需要训练** —— 新客户上线不需要等待几十秒到几百秒的 k-means 过程。
2. **码本可公开分发** —— 因为它不带任何客户统计信息,不会泄露隐私。
3. **同字节预算下召回更高** —— 4 个 Lloyd-Max 重建电平 + 1 bit 旋转 + 长度重归一化,比 2 个电平的符号阈值 (binary_quantize) 更靠近 Shannon 失真率下界。

所以 pg_turbovec 在 2-bit 模式下,Recall@10 = 1.000;同字节预算的 pgvector 1-bit Hamming HNSW,文献值大约 0.65~0.75。

  

## 三、3 种索引类型,3 种适用场景

pg_turbovec 不是单一工具,它提供 3 种索引类型,各有明确的适用场景。

**第一种:flat 平铺扫描** —— 用 `CREATE INDEX ... USING turbovec (embedding)` 默认创建。本质上是对压缩后的向量做 O(n·dim) 的 SIMD 全扫描。

在 1M × 1536 的真实数据上,AVX2 暖 p50 大约 2.5 秒。比 pgvector HNSW 慢 **大约 490 倍**。

听起来像是个灾难?但这种模式适合两个场景:
- 小规模精确召回 (< 10 万条,且不能容忍任何召回损失)
- 作为 IVF / Vamana ANN 的**精确 re-rank 阶段**

**第二种:IVF (倒排文件)** —— 用 `WITH (lists = N)` 启用。它做 cell-prune 扫描,并且**支持 out-of-core 构建**,索引可以比内存还大。

1M × 1536 实测暖 p50 大约 17~24 毫秒,Recall@10 在 0.84~0.89。这是**生产默认配置**。

**第三种:Vamana 图** —— 用 `WITH (graph = true)` 启用。DiskANN 风格的可导航图 ANN,面向低延迟场景。

另外 pg_turbovec 还提供一个非常重要的能力: **在 SIMD 内核里做白名单过滤**。这意味着对租户/类目 ID 做 WHERE 过滤时,选择性越强反而越快 —— 实测 crossover 大约在 7%~10% 选择性。这和 pgvector 默认的 post-filter 是完全相反的工程哲学。

  

## 四、看到「不损失召回」宣传,你需要立刻问的 4 个问题

pg_turbovec 的 README 和论文里到处是「不损失召回」「精确 re-rank」之类的字眼。但读得仔细一点,你会发现几个**必须老实面对的边界**。

**1) 老实说:flat 不擅长延迟**

1M 行 flat 暖 p50 大约 2.5 秒,比 pgvector HNSW 慢 **约 490 倍**。

这意味着:如果你看到有人说「pg_turbovec 比 pgvector HNSW 慢」,那个人大概率**在用 flat 模式跑生产**。这是错的用法。要生产里用 pg_turbovec,你必须用 IVF 或 Vamana,flat 仅适合做精确 re-rank。

**2) 缺 halfvec / sparsevec / bitvec**

pg_turbovec 只支持 f32 到标量量化,**不覆盖** halfvec、sparsevec、bitvec。

这意味着:如果你的工作流是 native binary embeddings (比如 Cohere 的 binary 模式)、图像指纹 (pHash / dHash 做近重复检测)、SimHash / MinHash 文本签名 —— 这些场景还是用 pgvector 的 HNSW + bit_hamming_ops 更合适。

**3) L2 / L1 ANN 缺位**

TurboQuant 内核打分的是**归一化向量的内积**。L2 和 L1 距离只能精确算,**没有 ANN 索引路径**。

如果你的工作流以欧氏 ANN 为主 (比如图像检索、3D 点云),pgvector + HNSW 仍然是正确选择。

**4) 不是 drop-in 替换**

pg_turbovec 的类型名是 `turbovec.vector`,schema 名是 `turbovec`。pgvector 用户的 `vector(1536)` 列需要显式通过 `::real[]::turbovec.vector` 迁移,无法零拷贝互转。

**二进制 varlena 兼容在 v1.0 路线图上, 但目前还不支持**。

  

## 五、那么,该用谁?

决策矩阵。

**应该选 pgvector + HNSW 的场景:**
- 内存预算充足,没有内存压力
- 需要 L2 / L1 ANN
- 需要 halfvec / sparsevec / bitvec
- 需要最成熟的生态和最大的客户端覆盖

**应该选 pg_turbovec 的场景:**
- 内存压力是核心瓶颈 (例如上亿条 1536 维嵌入)
- 多租户 SaaS 化向量索引,需要「过滤越强越快」的工程优势
- 对码本级别的隐私泄露有合规顾虑
- 工作流以 cosine / inner-product 为主,可以用 IVF / Vamana 模式跑生产

**应该选 pgvectorscale 的场景:**
- 语料规模到 5000 万条以上
- 可以运行 TimescaleDB
- 需要 StreamingDiskANN 这种为大规模优化的方案

  

## 六、最后 

pg_turbovec 不是银弹。它的 README 里有一句话, 我认为是整个 Postgres 生态里**最诚实的工程自我评价**之一: 

> "Use pg_turbovec when your workload is cosine / inner-product semantic search that is storage-constrained and you want exact-recall re-ranking and Postgres-native ACID/joins — and you use the IVF or graph kind (not a bare flat scan) for latency at scale. If you need raw HNSW latency, L2/L1 ANN, halfvec/sparsevec, pgvector + HNSW is the right pick today."

它的意思翻译过来就是: **这个工具不为所有人设计, 但它为那些真正需要它的人提供了一个非常稀缺的选项**。

PostgreSQL 向量检索的下一个 5 年,大概率会出现「多种索引共存」的格局 —— pgvector、pgvectorscale、pg_turbovec 各占一片山头。选哪个, 取决于你的内存预算、过滤场景、和延迟要求。

下次有人跟你说「我们在 Postgres 里存了 1 亿条向量,内存只用了 80 GB」,你可以问一句:

> "你们用的是 pgvector + HNSW,还是 pg_turbovec 4-bit? 如果是后者,你们生产用的是 IVF 还是 Vamana?"

—— 答得上来, 说明真的想清楚了; 答不上来,大概率在裸跑 flat 模式, 在生产里偷偷烧 CPU。
